# Qwen AutoGUI 架构逻辑与数据流

本文档记录了系统的核心逻辑架构，特别是"感知（Perception）"、"日志（Journaling）"与"自我进化（Self-Evolution）"的闭环流程。

---

## 1. 核心组件 (Core Components)

| 组件 | 作用 | 存储位置 |
| :--- | :--- | :--- |
| **Session** | 实时对话历史，包含聊天消息、`visual_log`（视觉旁白）、`debug_log` 和 rolling summary。 | `data/sessions/*.json` |
| **User Profile** | **核心记忆层**：包含 Objective (身份事实) 与 Subjective (主观规则) 记忆。 | `data/user_profile.json` |
| **Scene Memory** | **片段记忆层**：提取的事实性长期记忆（海量归档，支持向量检索）。 | `data/scene_memory.jsonl` |
| **Journal** | 每日总结日志，是进化的"原料库"（由每日汇总生成，不再实时写入）。 | `data/journals/YYYY-MM-DD.md` |
| **Diary** | AI 的第一人称感悟，是进化后的"成品"。 | `data/diary/YYYY-MM-DD.md` |
| **Interaction Habits** | AI 的行为习惯沉淀，由进化模块动态追加/修正，不修改 Persona 本体。 | `data/interaction_habits.md` |
| **Persona** | AI 的性格定义（只读基础人设，不会被自动修改）。 | `data/identities/*.md` |
| **Knowledge Graph** | 实体关系三元组图谱，支持 `query_knowledge` 工具查询。 | `data/knowledge_graph.jsonl` |

## 2. 数据流循环 (The Data Loop)

### A. 实时感知与存储阶段 (Perception & Storage)

视觉感知与聊天记录统一存入 **Session**，形成完整的、带时间戳的交互时间线。

**视觉感知 (`ContextManager`)**:
1. 后台线程每 **5 分钟**截屏一次，由视觉模型（`vision.model`，默认 `qwen3-vl-flash`）分析用户状态。
2. 注意：历史上下文注入逻辑（最近聊天记录 + 视觉旁白）目前在代码中已注释，视觉感知为**独立无上下文**分析。
3. 分析结果（状态 + 简评）以 `role = "visual_log"` 写入**当前的 `session.json`**，与聊天消息存在一起。
4. 若处于"活泼模式"且用户非工作状态，可能触发主动通知推送。

**聊天模型 (`Agent`)**:
1. 每次回复前，从当前 Session 中提取最近 **200 条**消息（`get_history(max_messages=200)`），过滤掉 `visual_log` 和 `debug_log`，不发送给 API。
2. **Omni-Context**：同时提取当前 Session 中最近 3 条 `visual_log`，注入到 System Prompt，让模型"感知"用户当前状态。
3. **Rolling Summary**：若 Session 存在 `summary` 字段（由滚动总结生成），会以伪对话形式注入到历史消息最前面，作为"前情提要"。

```
Session JSON 时间线示例:
[15:00] user: "帮我看看这段代码"
[15:01] assistant: "这里有个缩进问题..."
[15:05] visual_log: "working — 用户在 VSCode 编写 Python 代码"
[15:10] user: "好了我去看会B站"
[15:11] visual_log: "entertainment — 用户在 Bilibili 观看视频"
```

### B. 上下文压力管理：滚动总结 (Rolling Summary)

触发点：当 Session 估算 token 数超过 `(context_window - max_tokens) * 0.8` 时自动触发。

1. 裁剪最旧的 20% 消息（至少保留 10 条）。
2. 用 `evolution_model` 将裁剪内容压缩为"前情提要"文本。
3. 将原始对话片段追加到当天 Journal（`journal.append()`）。
4. 更新 `session.summary`，下次 `get_history()` 时自动注入。

这确保了超长对话不会因 context 溢出而崩溃，同时关键信息不丢失。

### C. 跨天/启动进化阶段 (Self-Evolution)

触发点：**程序启动时**（发现近 7 天内有未处理日期）或**运行时检测到跨天**（`_check_all_triggers`）。

#### 步骤 1：对话汇总 (`_summarize_day_conversations`)
1. 扫描所有 `data/sessions/*.json`，提取指定日期的全部消息（含 `visual_log`，排除 `debug_log`）。
2. **按时间戳排序**，合并多个 Session 为完整的单日时间线。
3. 格式化时，`visual_log` 转换为 `【系统旁白:视觉感知】` 角色，与对话穿插排列；多模态图片内容替换为 `[图片附件]` 占位符。
4. 调用 LLM 将混合时间线压缩为主题总结，追加写入 `data/journals/YYYY-MM-DD.md`。

此时 Journal 包含了语言与行为交织的完整一天记录。

#### 步骤 2：日志进化 (`evolve_from_journal`)
`SelfEvolution` 模块读取完整的 Journal，串行执行：
1. **合成日记 (Diary)**：以 AI 第一人称视角写下一天的感悟，写入 `DiaryManager`。
2. **分层记忆提取 (`_extract_memories`)**：
   - **User Profile (Core)**：通过 `profile_updates` 字段，直接调用 `user_profile.update_objective/update_subjective()` 固化用户属性与规则。
   - **Scene Memory (Episodic)**：通过 `memories` 字段，调用 `memory.add/update()` 写入片段记忆库。
   - 提取前会注入现有记忆库摘要（最近 50 条）和 User Profile 作为上下文，避免重复提取。
3. **知识图谱 (`_extract_triples`)**：提取实体关系三元组，写入 `KnowledgeGraph`。
4. **主动探索 (`explore_curiosities`)**：从 Journal 中挖掘当天未解决的疑问与感兴趣的概念，自主联网研究后将成果蒸馏入记忆库与知识图谱（详见 Section 2D）。

#### 步骤 3：人设微调 (`evolve_persona`)
读取最近 20 条记忆，评估是否需要追加或修正 `data/interaction_habits.md`。
修改前自动通过 `PersonaAudit` 保存快照，支持回滚。

### D. 主动探索与知识沉淀 (Agentic Curiosity Exploration)

这是进化流程的第四步，在记忆提取和知识图谱之后、人设微调之前执行。

**触发条件**：每次 `evolve_from_journal` 执行时自动运行（每日一次）。

**执行流程**：

```
Journal
  └─ [CURIOSITY_EXTRACT_PROMPT]
       └─ LLM 分析 → 疑问列表（最多 3 条/天）
            └─ 对每条疑问：
                 ├─ 调用主模型 + enable_search=True 联网研究
                 ├─ [RESEARCH_DISTILL_PROMPT] 蒸馏研究结果
                 ├─ worth_remembering=true → memory.add(tags=["agentic-research"])
                 ├─ triples → kg.add_batch(source="research:日期:主题")
                 └─ 研究摘要追加写入当天 Journal（供后续查阅）
```

**疑问筛选标准**（`CURIOSITY_EXTRACT_PROMPT`）：
- 对话中出现"不知道"、"不确定"等模糊表述背后的知识点
- 提到但未深入展开的技术概念或工具
- 解决问题时绕过了某个知识盲区（用了 workaround 但未搞清根本原因）
- 值得长期了解的领域动态

**防护机制**：
- 每天最多处理 3 个疑问，避免 API 过度消耗
- 每次搜索之间 sleep(2) 防止频率限制
- 研究和蒸馏步骤均有独立异常捕获，单条失败不影响其他条目
- 记忆写入时自动附加 `agentic-research` 和日期标签，便于溯源

### 跨天检测触发器 (`agent.py`)
```python
# _startup_evolution (启动时，延迟 10 秒执行)
# 向前扫描最多 7 天，找到[有 session 消息但没日记]的日期
# -> 串行执行：_summarize_day_conversations -> evolve_from_journal -> evolve_persona
# -> 每个日期处理完后 sleep(5) 防止 429

# _check_all_triggers (每次 chat/chat_stream 结束后 + 后台每 60 秒)
# if current_date != last_interaction_date:
#    后台线程 -> 对昨天日期执行汇总与进化，不阻塞当前问答
# if est_tokens > SAFE_LIMIT:
#    _consolidate_rolling() -> 滚动总结
```

### 记忆分层与注入机制
1. **Objective Memory (客观档案)**：[常驻注入 System Prompt] 姓名、年龄、硬件环境等。
2. **Subjective Memory (主观偏好)**：[常驻注入 System Prompt] 用户的"戒律"与习惯。
3. **Scene Memory (场景事实)**：[Top-K 向量检索] 根据当前 query 语义相似度动态注入最相关的 3 条。
4. **Interaction Habits**：[常驻注入 System Prompt] 从文件读取，有文件变更时自动热重载（mtime 检测）。

**即时偏好更新**：当用户说"以后注意..."时，模型立即调用 `update_user_profile` 工具更新档案，无需等到跨天进化。

### 多模态图片处理路径
- 若 `image_analyzer` 配置与主模型**不同**（Vision Proxy 模式）：先用专属视觉模型解析图片为文字描述，再将描述交给主模型处理，主模型保留完整 Tool 权限。
- 若 `image_analyzer` 配置与主模型**相同**（Omni-Modal 模式，如 `qwen3.5-plus`）：主模型直接处理多模态输入，无需代理。

## 4. 手动干预与运维工具

根目录提供 `force_evolve.py` 脚本，可手动强制触发某一天的全部汇总流程。

```bash
# 删除已有日记（允许重新生成）
del data\diary\2026-02-19.md

# 清空片段记忆与图谱（可选，彻底重置）
del data\scene_memory.jsonl
del data\scene_memory_vectors.jsonl
del data\knowledge_graph.jsonl

# 重新触发进化
python force_evolve.py 2026-02-19
```

人设快照回滚（通过 `PersonaAudit`）：
```python
from core.persona_audit import PersonaAudit
audit = PersonaAudit(persona_path="data/interaction_habits.md", data_dir="data")
print(audit.diff(0))   # 查看与最旧快照的差异
audit.rollback(0)      # 回滚到指定快照
```

## 5. 模型生态与功能分布 (Model Ecology)

系统中同时运行着多个不同定位的 AI 模型，它们各司其职构建了完整的架构：

| 模型分类 | 配置文件项 | 核心作用与代码调用位置 |
| :--- | :--- | :--- |
| **主回复模型**<br>(Chat Model) | `api.model` | **主管前台业务交互**：<br>- `agent.py -> chat() / chat_stream()`: 处理日常对话，解析并调用 Tools/Skills。<br>- `agent.py -> _summarize_day_conversations()`: 跨天时对历史对话+视觉日志进行压缩汇总，生成原始 Journal。<br>- `agent.py -> _consolidate_rolling()`: 上下文压力触发时执行滚动总结。 |
| **视觉感知模型**<br>(Vision Model) | `vision.model`<br>*(后台截屏分析专用)* | **主管后台环境感知**：<br>- `context.py -> analyzer_loop()`: 独立后台线程每 5 分钟截屏，分析用户行为状态，产出 `visual_log`。 |
| **图片解析模型**<br>(Image Analyzer) | `image_analyzer.model`<br>*(用户上传图片解析)* | **主管用户主动上传的图片**：<br>- 若与主模型相同（全模态如 `qwen3.5-plus`）：直接由主模型处理，无代理开销。<br>- 若不同：Vision Proxy 模式，先解析为文字再交主模型，保留完整 Tool 权限。 |
| **进化模型**<br>(Evolution Model) | `api.evolution_model`<br>*(默认同主模型)* | **主管离线人格整理**（集中在 `core/self_evolution.py`）：<br>- `evolve_from_journal()`: 写日记、提取 User Profile 更新、提取 Scene Memory。<br>- `_extract_triples()`: 提取知识图谱三元组。<br>- `evolve_persona()`: 反思并修改 `interaction_habits.md`。 |
| **向量模型**<br>(Embedding Model) | `embedding.model` | **主管记忆的语义检索**：<br>- `vector_memory.py`: 存储、搜索 Scene Memory 时计算高维嵌入。<br>- `agent.py -> _backfill_vectors()`: 后台补全历史记忆向量。<br>- `journal_index.py`: 对历史 Journal 做向量化索引，支持 `search_journal` 语义搜索。 |

## 6. 技能与插件体系 (Skill & Plugin System)

系统通过三层结构扩展能力，支持从硬编码工具到动态文档自学的全过程：

### A. 内置技能 (Built-in Skills)
- **定义位置**：`core/agent.py -> _register_builtins()`。
- **核心工具**：
    - `remember` / `search_memory`: 长期记忆入库与三合一检索（Journal + Diary + Scene Memory + KG 并发查询）。
    - `update_memory` / `delete_memory`: 记忆的修正与删除。
    - `update_user_profile`: 实时修改用户档案（objective/subjective 两层）。
    - `query_knowledge`: 查询知识图谱实体关系。
    - `new_session` / `list_sessions`: 会话管理。
- **计划工具**（由 `plugins/plan_handler.py` 注册）：`create_plan` / `update_plan_step` / `complete_plan`。

### B. 插件体系 (Plugin System)
- **管理器**：`core/plugin_manager.py (PluginManager)`。
- **加载逻辑**：启动时扫描 `plugins/*.py`，调用其 `register(skill_manager)`。
- **热更新**：支持运行时 `/plugins reload` 重新加载 Python 逻辑。
- **生命周期**：插件可追踪并清理自己注册的 Skill，防止卸载后残留。

### C. 文档驱动技能 (SKILL.md / Documentation Skills)
这是最灵活的扩展方式，允许 AI 通过阅读 Markdown 文档来学习如何使用外部 CLI 工具。

1. **扫描与索引**：
    - `agent.py -> _scan_local_skills()`: 扫描 `skills/` 和 `.agents/skills/` 目录。
    - 提取 YAML Metadata（名称、描述、`allowed-tools` 白名单）及 `scripts/` 子目录脚本。
2. **意图匹配与引导**：
    - `agent.py -> _find_relevant_skills(user_query)`: 每次对话时分词匹配描述，命中后主动在提示词中建议用户尝试该技能。
3. **按需学习 (Progressive Disclosure)**：
    - `list_skills`: 向 LLM 展示简要目录及描述。
    - `get_skill_info(name)`: LLM 调用此工具读取完整的 `SKILL.md`，实现"现学现卖"。
4. **动态安装**：
    - `install_skill(url)`: 从 Git 或 URL 下载、解析、存储并注册新技能，自动生成 `plugins/remote_*.py` 包装器。

## 7. 辅助模块 (Supporting Modules)

### MCP 客户端 (`core/mcp_client.py`)
实现了轻量级异步 JSON-RPC 2.0 客户端，通过 stdio 与外部 MCP Server 通信。

- **协议**：MCP 2024-11-05，支持 `initialize` 握手、`tools/list` 发现、`tools/call` 执行。
- **通信模式**：支持 exec 模式（命令+参数列表）和 shell 模式（字符串命令）。
- **并发**：基于 `asyncio`，使用 `Future` 字典管理 pending 请求，后台 reader task 异步解析响应。
- **错误处理**：超时（默认 30s，工具调用 60s）、进程退出、JSON 解析失败均有处理。
- **用途**：允许系统接入任意符合 MCP 协议的外部工具服务器，扩展 AI 能力边界。

### 日记向量索引 (`core/journal_index.py`)
对历史 Journal 文件做语义向量化，支持跨日期的语义搜索。

- **存储**：`data/journal_vectors.jsonl`，每行一个 chunk（含 date、chunk_idx、text、vector）。
- **分块**：调用 `vector_memory.split_text(chunk_size=600, overlap=60)` 切分。
- **检索**：对所有 chunk 计算余弦相似度，按日期做 max-pooling，返回 Top-K 日期及片段。
- **去重**：`has_indexed(date_str)` 防止重复索引同一天。
- **用途**：支撑 `search_memory` 工具中的 Journal 语义搜索分支。

### 人设审计与快照 (`core/persona_audit.py`)
为 `interaction_habits.md` 提供版本快照与回滚能力。

- **快照**：每次 `evolve_persona` 修改前自动调用 `snapshot(reason=...)`，保存到 `data/persona_snapshots/YYYY-MM-DD_HH-MM-SS.md`。
- **对比**：`diff(idx)` 展示当前文件与指定快照的行级差异（增/删行）。
- **回滚**：`rollback(idx)` 恢复到指定快照，回滚前再次保存当前状态作为备份。
- **注意**：`PersonaAudit` 现在管理的是 `interaction_habits.md`，而非 `PERSONA.md`（Persona 文件本身不可变）。

## 8. 维护说明
- **视觉感知频率**：调整 `main.py` 中的 `interval_seconds`（默认 300 秒）。
- **主动交互模式**：运行时输入 `/mode` 切换静默/正常/活泼模式，或在前端配置面板调整。
- **记忆提取逻辑**：修改 `self_evolution.py` 中的 `EXTRACTION_PROMPT`。
- **彻底重置某日**：删除对应的 `data/diary/YYYY-MM-DD.md` 并重新运行 `force_evolve.py`。
- **上下文窗口配置**：在 `config.json -> api.context_window` 设置模型总 context 长度（默认 128000），影响滚动总结触发阈值。
- **多人设管理**：人设文件存放于 `data/identities/*.md`，通过 `agent.switch_persona(name)` 切换，配置持久化到 `data/config.json`。

## 9. 3D & VRM 进阶系统逻辑

### A. 视口持久化与热切换 (Persistence & Seamless Switching)
1. **Snapshot 机制**：在 `switchVrmModel` 前，系统会捕捉当前相机的 `position`、`quaternion` 以及模型自身的 `position`/`scale` 存入快照。
2. **RAF 延迟应用**：由于 `loadModel` 内部会重置相机，系统使用 `requestAnimationFrame` 延迟 2 帧，在渲染稳定后再将快照参数强行覆盖，实现视觉上的"无缝"继承。
3. **动画遗留继承**：切换模型后，系统会自动读取 `vrm-animation.js` 中记录的 `_lastVrmaUrl` 并重播，保持动作连贯性。

### B. VRM 商店与资源下载流 (Store & Async Download)
1. **资源感知**：
   - 模型目录：`static/models/*.vrm`
   - 动画目录：`static/vrm/animation/*.vrma`
2. **下载链路**：
   - 前端发起 `POST /api/store/download`。
   - 后端 `server.py` 使用 `httpx` 异步流式写入文件至对应目录。
   - 前端通过列表差异比对（如 `isAnimationDownloaded`）实时更新商店 UI 为"已下载"状态，并动态刷新全局快捷动作按钮。
